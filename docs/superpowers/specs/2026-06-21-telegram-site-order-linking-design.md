# Telegram Site Order Linking Design

## Goal

Сделать Telegram-бот единым клиентским кабинетом GPTishka, где пользователь видит все свои покупки: оформленные на сайте и оформленные внутри Telegram-бота.

Клиент не должен терять данные после покупки. Если он купил VPN или подписку, потом вернулся через 10–20 дней и открыл бота, он должен видеть заказ, статус, срок действия, инструкции и доступные данные для получения услуги.

## Current State

Сейчас в проекте уже есть:

- сайт с checkout и модальным окном заказа;
- backend-заказы, оплаты, промокоды, CDK, VPN-выдача и активации;
- Telegram worker;
- Telegram-заказы с `telegramUserId`, `telegramChatId`, `telegramUsername`;
- команды `/start`, `/buy`, `/orders`, `/token`, `/check`;
- отдельный Telegram-магазин, который создаёт заказы через общий backend.

Не хватает безопасного моста:

```text
заказ создан на сайте
→ клиент нажал "Получить данные в Telegram"
→ бот связал этот заказ с Telegram user id
→ заказ появился в "Мои покупки" вместе с Telegram-заказами
```

## Product Rule

Все покупки клиента должны отображаться вместе.

Не должно быть разделения:

- "заказы с сайта";
- "заказы из Telegram";
- "VPN отдельно";
- "активации отдельно".

Для клиента это один раздел:

```text
Мои покупки
```

Внутри списка должны быть все активные и последние заказы клиента, независимо от места оформления.

## Telegram Limitation

Бот не может первым написать пользователю только по `@username`.

Чтобы бот мог отправлять данные клиенту, пользователь должен хотя бы один раз открыть бота и нажать `/start`.

Поэтому поле Telegram в модальном окне сайта используется как контакт, но техническая привязка делается только через deep-link:

```text
https://t.me/GPTishka_myBot?start=order_<orderId>_<token>
```

## Site Checkout Flow

После создания оплаты backend уже возвращает:

- `order_id`;
- `activation_token`;
- `activation_url`;
- `pay_url`.

Нужно добавить клиентский сценарий:

1. После оплаты или на странице ожидания показать кнопку:

```text
Получить данные в Telegram
```

2. Кнопка открывает:

```text
https://t.me/GPTishka_myBot?start=order_<orderId>_<activationToken>
```

3. Если клиент не нажал кнопку, заказ всё равно остаётся в системе и виден менеджеру в админке.

## Bot Linking Flow

Когда бот получает:

```text
/start order_<orderId>_<token>
```

он должен:

1. Найти заказ по `orderId`.
2. Проверить `token` через существующий `redeemTokenHash`.
3. Проверить, что заказ не привязан к другому Telegram user id.
4. Если заказ уже привязан к этому же user id — считать это повторным входом, не ошибкой.
5. Сохранить в заказ:
   - `telegramUserId`;
   - `telegramChatId`;
   - `telegramUsername`;
   - источник привязки: site deep-link.
6. Отправить клиенту понятный статус заказа.

Если в заказе с сайта был указан `@username`, можно сравнить его с username Telegram-профиля, но нельзя жёстко блокировать заказ только из-за несовпадения. Username можно изменить, удалить или указать с ошибкой.

## Unified "My Purchases"

Команда:

```text
/orders
```

и кнопка:

```text
Мои покупки
```

должны показывать все заказы, где:

- `telegramUserId` совпадает с текущим пользователем;
- заказ может быть создан в Telegram;
- заказ может быть создан на сайте и позже привязан через deep-link.

Список должен быть удобным для клиента, который покупает разные услуги:

```text
1. GPTishka VPN — активен, осталось 10 дней
2. ChatGPT Go — требуется токен для активации
3. Claude Pro — активируется менеджером
4. SuperGrok — активирован
```

## Data Display Rules

### VPN

Для VPN бот должен показывать:

- название продукта;
- статус;
- срок действия;
- сколько дней осталось;
- VLESS-ключ;
- кнопку "Скопировать ключ" невозможно сделать нативно в Telegram, поэтому ключ должен быть отдельным моноширинным блоком;
- инструкцию или ссылку на инструкцию;
- кнопку "Продлить".

### ChatGPT / Claude / SuperGrok

Для подписок бот должен показывать:

- название продукта;
- тариф;
- способ: со входом / без входа / id / token / ручная обработка;
- статус оплаты;
- статус активации;
- ссылку на страницу активации, если клиент должен ввести токен/id;
- понятный текст, если заказ в ручной обработке.

### Sensitive Data

Секретные данные нельзя бездумно слать в общий админ-чат.

В личный чат клиента можно отправлять:

- VLESS-ключ;
- ссылку на активацию;
- статус заказа;
- инструкцию.

Пароли/токены клиента лучше не повторять в Telegram, если это не требуется для оказания услуги. В админке они должны быть доступны через контролируемое отображение.

## Customer-Facing Error Policy

Клиент не должен видеть технические ошибки вроде:

- provider timeout;
- CDK missing;
- activation failed;
- webhook mismatch;
- no key in pool.

Клиенту показывается мягкое состояние:

```text
Оплата получена.
Заказ принят в работу.
Если автоматическая выдача заняла больше обычного, менеджер уже видит задачу.
```

Техническая причина должна быть видна в админке и/или менеджерском Telegram-уведомлении.

## Manager Notifications

Если заказ не может быть выполнен автоматически, бот/система должны отправить менеджеру:

- order id;
- товар;
- тариф;
- email;
- Telegram;
- причина остановки;
- ссылка на заказ в админке.

Сообщение менеджеру не должно содержать полный пароль/токен клиента, если это не личный защищённый канал и не требуется прямо сейчас.

## Security Rules

1. Нельзя привязать заказ к Telegram без `orderId + token`.
2. Нельзя активировать заказ без статуса `PAID`.
3. Нельзя повторно привязать заказ к другому Telegram user id без ручного действия в админке.
4. Нельзя выдавать CDK/VPN по одному `@username`.
5. Deep-link token должен использовать существующую логику `redeemTokenHash`.
6. Public status не должен отдавать CDK, VLESS или пароли без подтверждения Telegram-связки или activation token.

## Non-Goals

В этой задаче не делаем:

- новый Telegram-магазин с нуля;
- новые цены отдельно для бота;
- отдельную базу товаров для бота;
- email-доставку данных;
- замену текущей системы оплаты;
- переписывание всего Telegram worker.

Telegram должен стать вторым интерфейсом к тем же данным, которые уже использует сайт.

## Success Criteria

Функция считается готовой, когда:

1. Клиент после покупки на сайте может нажать кнопку "Получить данные в Telegram".
2. Бот по `/start order_<orderId>_<token>` привязывает заказ к Telegram user id.
3. Привязанный сайтовый заказ появляется в `/orders`.
4. Telegram-заказы и сайт-заказы отображаются в одном списке.
5. VPN-заказ показывает клиенту действующий ключ и срок.
6. Заказы ChatGPT/Claude/SuperGrok показывают понятный статус и ссылку/инструкцию.
7. Заказ нельзя привязать к чужому Telegram без token.
8. Заказ нельзя получить по одному `@username`.
9. Если автоматическая выдача невозможна, клиент видит мягкий статус, а менеджер видит техническую причину.
